django standalone

앞선 포스팅에서 네이버 블로그에 있는 벤큐 피시방 목록을 크롤링 하는데에는 성공했다. 이제 크롤링한 정보를 db에 pcroom모델 객체로 저장해야 된다. pcroom의 모델은 다음과 같다.

manage.py shell 에서 하듯이 django orm을 이용해 pcroom 모델을 임의로 생성해보았다.

이제 script.py 를 실행하면 PCroom 객체가 하나 생성되고 db에 저장되야 하지만

대강 이런 에러가 뜬다. script.py에 장고 환경 변수가 설정이 안되있다는 뜻이다. 이를 해결하기 위해 django standalone이 있다. django standalone은 일반 파이썬 스크립트에서도 ORM 같은 장고 기능을 사용할 수 있게해준다. django standalone을 사용하기 위해서는 파일 상단에 다음 4줄만 추가해 주면 된다.

django standalone을 몰랐을 때는 sqlite db에 직접 접근해서 쿼리문으로 PCroom객체들을 저장했다. 잘 작동했지만 문제가 있었다. 배포할 때는 postgres같은 다른 db를 사용해야 하는데 sqlite 쿼리문으로는 데이터를 저장할 수 없는 것이다. django standalone은 외부 파일에서 ORM말고 다른 장고 기능을 사용할 때에도 요긴하게 쓸 수 있을것 같다.

Crawler-02-Selenium

타겟 : https://benq4ever.blog.me/221196878240
셀레니움으로 네이버 블로그 크롤링 하기

https://sites.google.com/a/chromium.org/chromedriver/downloads
셀레니움을 사용하기 위해선 먼저 드라이버를 설치해야한다. 위 링크에서 다운로드 받은 파일을 프로젝트 폴더에 넣어주자.

먼저 셀레니움으로 타겟의 html 소스를 가져와봤다. 포스트의 본문은 아래 frame에 숨겨져있다. frame에 접근하기위해 switch_to.frame() 메서드를 사용한다. 함수 인자는 frame의 id, 또는 frame의 위치도 가능하다.(이 경우 0)

새로운 html소스로 보아 frame 접근에 성공했음을 알 수 있다. 다만, 이번에도 포스트의 본문은 보이지 않고 iframe이라는 새로운 tag가 두개나 등장했다. mainFrame과 hiddenFrame. 본문이 들어있을 법한 이름을 가진 mainframe에 접근해보자. screenFrame 접근 방식과 동일하다.

html 소스를 출력해보면 본문이 포함되 있음을 알 수 있고, 이제 필요한 정보를 골라내기 위해 BeatifulSoup를 사용해야한다.

‘div#postViewArea div.view table:nth-of-type(2)’는 css selector의 문법이다. 나는 블로그 포스트의 표를 가져오기 위해 이렇게 작성했지만, 포스트의 다른 부분을 가져와야 한다면 수정해야한다.

찾아보니 frame과 frameset은 html5에서 더이상 지원하지 않는 퇴화 태그라고 한다. 왜 네이버 블로그는 이런 퇴화 태그를 써가면서 복잡하게 설계한걸까? 구글같은 경쟁 사이트에 노출이 어렵게 하기 위함이라는 의견도 보았으나 잘 모르겠다. 어쨌거나 네이버 블로그는 frame을 하나씩 열어가며 크롤링 해야한다는 것을 알 수 있었다. 다음 포스트에서는 크롤링한 정보를 django 모델로 저장하여 데이터베이스에 반영해 보겠다.

네이버 블로그 크롤링하기 01/

이번 프로젝트의 목표는 네이버 블로그 포스트 본문을 긁어오는 게 목표이다
파이썬으로 크롤링을 하기 위해서는 두 가지 도구가 필요하다.

  • requests
  • BeatifulSoup

requests는 말 그대로 요청이다. 브라우저가 서버에 요청하듯 파이썬을 이용해 해당 url의 정보를 get해온다.

BeatifulSoup는 requests를 이용해 가져온 html 소스에서 원하는 정보를 탐색하기 용이하게 해준다.

일단 requests를 이용해 위키백과의 파이썬 항목의 본문을 크롤링 해보자.

문제는 네이버 블로그다. 네이버 블로그 포스트를 위와 같은 방법으로 크롤링 해보자.

블로그 이름만 보일 뿐 본문은 찾을 수 없다. 포스트의 본문은 소스 하단의 frame에 저장되어 있기 때문이다. requests로는 frame에 접근할 수 없다. 따라서 다른 도구를 사용해야 한다. 다음 포스트에서 selenium이라는 도구를 활용해 frame에 접근해보자.
